## 3. 服务架构

```

    │
    ├── 静态页面 → Vite Dev Server (node)
    └── /api/* → Go API Server (:8080)
                    │
                    ├── MySQL (:3306) — 业务数据持久化
                    ├── Redis (:6379) — 探针在线状态缓存 + 工作流执行缓存

                    │       └── Node Exporter (:9100) — 主机指标采集

                    └── 外部 API（可选）
```

## 5. 日常运维操作

### 5.1 启动全部服务

```bash
# 方式一：使用启动脚本
bash /opt/ai-workbench/start-wsl.sh

# 方式二：手动逐个启动
sudo systemctl start mysql redis-server prometheus-node-exporter
cd /opt/ai-workbench/api && nohup ./api-linux >> /opt/ai-workbench/logs/api.log 2>&1 &
prometheus --config.file=/opt/ai-workbench/docker/prometheus.yml \
  --storage.tsdb.path=/opt/ai-workbench/prometheus-data \
  --web.listen-address=0.0.0.0:9090 --web.enable-lifecycle &
cd /opt/ai-workbench/web && npm run dev -- --host 0.0.0.0 --port 3000 &
```

### 5.2 停止全部服务

```bash
bash /opt/ai-workbench/stop-wsl.sh
```

### 5.3 重启 API（代码更新后）

```bash
# 编译
cd /opt/ai-workbench/api && go build -o api-linux .

# 重启
pkill -f api-linux
nohup ./api-linux >> /opt/ai-workbench/logs/api.log 2>&1 &
```

### 5.4 查看日志

```bash
# API 日志
tail -f /opt/ai-workbench/logs/api.log

# MySQL 日志
sudo journalctl -u mysql -f

# Redis 日志
sudo journalctl -u redis-server -f
```

### 5.5 数据库备份

```bash
mysqldump -u root -p'<YOUR_PASSWORD>' -h 127.0.0.1 ai_workbench > /opt/ai-workbench/backup_$(date +%Y%m%d).sql
```

### 5.6 数据库恢复

## 9. 安全注意事项

- MySQL 已设置 root 密码（mysql_native_password 认证），Redis 当前无密码，均仅监听 127.0.0.1
- 凭据（credentials 表）中的密码当前明文存储，敏感环境需加密
- Admin Token 未配置时为 permissive 模式，生产环境必须设置 AIW_ADMIN_TOKEN
- SSH 远程执行使用 InsecureIgnoreHostKey，生产环境建议配置 known_hosts
- 远程主机白名单在 security/guard.go 中硬编码，需按实际环境修改

## 10. 原生知识库与诊断管理

### 10.1 原生能力说明

知识库、案例库、诊断工作流均由 Go API 内置实现，依赖 MySQL、Redis、Prometheus 与 LLM 配置，不需要额外 sidecar 服务。

### 10.2 知识库管理（案例库）

**进入「知识库」页面**：
- 案例分类：cpu_high / memory_leak / disk_full / oom / network_issue / connection_pool 等
- 支持关键词搜索（FULLTEXT 索引）和分类筛选
- 「新建案例」/「编辑」/「删除」单条操作
- 「导入」上传 JSON 数组批量创建
- 「导出」下载全量 JSON

**冷启动**：导入种子案例（18 条覆盖 12 个故障场景）：
```bash
curl -X POST -H 'Content-Type: application/json' \
  -d @api/assets/seed_cases.json \
  http://localhost:8080/api/v1/knowledge/cases/import
```

### 10.3 运维手册（Runbooks）

**进入「运维手册」页面**：
- 标准化故障处置流程（SOP），按分类管理
- trigger_conditions 是结构化 JSON，便于后续告警自动匹配
- steps 是 Markdown 格式，包含可执行命令
- auto_executable 标识是否允许自动执行（默认 false）

**冷启动**：导入 10 条 Runbook 种子数据：
```bash
for rb in $(jq -c '.[]' api/assets/seed_runbooks.json); do
  curl -X POST -H 'Content-Type: application/json' \
    -d "$rb" http://localhost:8080/api/v1/knowledge/runbooks
done
```

### 10.4 指标映射管理

**进入「指标映射」页面**，标准操作流程：

1. **扫描指标**：调用 Prometheus `/api/v1/label/__name__/values` 获取全量指标名（首次扫描通常 500+ 条）
2. **AI 自动适配**（开发中）：将 unmapped 指标分批发给 LLM，返回标准名 / exporter / 描述 / PromQL，写入数据库（status=auto）
3. **用户确认**：单条编辑或批量确认 AI 结果（status=auto → confirmed）
4. **本地生效**：确认后的映射写入 `metrics_mappings`，供诊断链路查询和报告生成

数据存储在 `metrics_mappings` 表，唯一约束 (datasource_id, raw_name)。

### 10.5 诊断工作流

**进入「诊断工作流」页面**：
- 输入主机名/IP + 时间范围 + 可选问题
- 响应模式：blocking 阻塞 / streaming 流式（推荐）
- 7 节点流水线：start → knowledge_retrieval → http_metrics → http_inspection → llm_diagnosis → condition → end
- 报告卡片：根因 + 证据表格 + 处置步骤 + 置信度（HIGH/MEDIUM/LOW）
- 反馈：准确 / 部分准确 / 不准确 → 写入本地反馈闭环
- 归档：将诊断结果归档为案例 → 进入本地案例库

### 10.6 反馈与归档闭环

**反馈**（POST /api/v1/diagnosis/feedback）：
- rating ∈ {accurate, partial, inaccurate}
- accurate → 记录为准确反馈，参与知识沉淀
- inaccurate → 记录为不准确反馈，进入改进样本
- partial → 记录为部分准确反馈，保留人工备注

**归档**（POST /api/v1/diagnosis/archive）：
- 将诊断记录的 raw_report 提取为 metric_snapshot
- 创建新案例，source_diagnosis_id 指向原诊断
- 归档失败时返回错误，不影响原始诊断记录

### 10.7 降级策略

| 场景 | 降级行为 |
|------|---------|
| 知识库检索为空 | 工作流跳过知识证据，继续采集指标并调用 LLM |
| Prometheus 不可用 | 指标扫描失败，工作流跳过 http_metrics 节点 |
| LLM 调用失败 | AI 自动适配返回 0 条，用户可手动编辑映射 |

### 10.8 告警驱动自动诊断

告警 webhook（POST /api/v1/alert/webhook 或 /api/v1/alert/catpaw）接收后：
1. 去重存入 alerts 表
2. 自动创建 DiagnoseRecord（trigger=alert）
3. 异步调用 RunDiagnose 进行 AI 诊断
4. 诊断完成后通过 ActionLog 关联回告警

可在「告警中心」页面查看告警关联的诊断记录。

### 10.9 诊断报告对比

GET /api/v1/diagnose/compare?ip=&limit=2 返回同主机最近 N 次诊断的 diff，便于排查问题是否复发。需要至少 2 次完成的诊断才能对比。

## 11. 原生工作流引擎

### 11.1 架构说明

原生工作流引擎所有能力编译进单一 Go 二进制，零额外运行时依赖。

```
YAML DSL 定义
    │
    ▼
dsl.go（解析 + embed.FS 加载）
    │
    ▼
graph.go（DAG 构建 + Kahn 拓扑排序 + 环检测）
    │
    ▼
engine.go（执行引擎）
    ├── 阻塞模式：同步执行，返回完整结果
    ├── 流式模式：SSE 事件推送（6 种事件类型）
    ├── 并行执行：parallel_group 节点（并行分支 + 汇聚等待）
    ├── 单节点重试：可配置重试次数和间隔
    └── 超时控制：全局 + 单节点超时
    │
    ▼
variable.go（线程安全变量池 + {{nodeID.field}} 插值）
    │
    ▼
streaming.go（SSE 事件发射器）
```

**20 种节点类型**：

| 分类 | 节点 |
|------|------|
| 基础 | start / end / condition / variable_aggregator / variable_assigner |
| LLM | llm / parameter_extractor / question_classifier |
| 数据 | knowledge_retrieval / document_extractor / http_request / tool |
| 流程 | loop / iteration / list_filter / template_transform |
| 编排 | sub_workflow / human_input |
| 代码 | code（goja JS 沙箱，13 种危险模式拦截） |
| 智能体 | agent（工具调用循环，最多 15 轮） |

**桥接层**（bridge.go）：

- `LLMBridge` — 注入 LLM 调用能力（复用 ai 配置段）
- `KnowledgeBridge` — 注入知识库检索能力（BM25 + 向量混合搜索）
- `ToolBridge` — 注入工具执行能力（HTTP 请求、Prometheus 查询等）

### 11.2 内置工作流（30 个：9 核心 + 21 别名）

**核心工作流（9 个）**：

| 工作流 | 文件 | 用途 |
|--------|------|------|
| 智能诊断 | smart_diagnosis.yaml | 知识检索 + 指标采集 + 告警采集 + LLM 根因分析 + Runbook 匹配 |
| 专项诊断 | domain_diagnosis.yaml | 根据 domain 参数选择容器、JVM、数据库、日志等专项链路 |

### 11.3 并行执行

引擎支持 `parallel_group` 节点类型，允许多个分支并行执行后汇聚：

```yaml
- id: parallel_check
  type: parallel_group
  data:
    branches:
      - [metrics_query, log_query]
      - [network_check]
    timeout_seconds: 60
```

所有分支完成后，结果合并到变量池，后续节点可通过 `{{parallel_check.branch_0.result}}` 引用。

### 11.4 缓存配置

工作流执行结果支持缓存，避免短时间内重复执行相同工作流。

```yaml
# config.yaml
workflow:
  cache:
    enabled: true       # 是否启用缓存
    ttl_seconds: 300    # 缓存有效期（秒），默认 5 分钟
```

缓存策略：
- 优先使用 Redis 缓存（分布式场景）
- Redis 不可用时自动降级到内存缓存
- 缓存 key 基于工作流 ID + 输入参数哈希
- 支持手动清除缓存（通过工作流管理 API）

## 12. 知识库文档管理

### 12.1 架构说明

知识库文档管理提供完整的文档生命周期：上传 → 解析 → 分块 → Embedding → 索引 → 搜索。

```
文档上传（MD/TXT/PDF/DOCX/HTML）
    │
    ▼
parser.go（格式识别 + 内容提取）
    │
    ▼
chunker.go（智能分块）
    ├── 按段落分块（Markdown 标题感知）
    ├── 按固定长度分块（可配置 chunk_size）
    └── 重叠窗口（overlap 防止语义断裂）
    │
    ▼
indexer.go（索引构建）
    ├── BM25 索引（bm25.go，中文分词 gse）
    └── 向量索引（vector.go，外部 Embedding API）
    │
    ▼
hybrid.go（混合搜索）
    ├── BM25 关键词匹配
    ├── 向量语义相似度
    ├── RRF 融合排序
    └── Reranker 二次排序（可选）
```

### 12.2 Embedding 配置

```yaml
# config.yaml
embedding:
  provider: "builtin"          # builtin（仅 BM25）或 api（BM25 + 向量）
  api:
    url: ""                    # 向量 API 地址（OpenAI 兼容接口）
    key: ""                    # API Key（通过环境变量设置更安全）
    model: "text-embedding-3-small"  # Embedding 模型
    dimensions: 1536           # 向量维度
    batch_size: 20             # 批量请求大小
```

- `provider: "builtin"` — 仅使用 BM25 关键词搜索，无需外部 API，零成本
- `provider: "api"` — BM25 + 向量混合搜索，需要 Embedding API（支持 OpenAI、Azure OpenAI 等兼容接口）

### 12.3 Reranker 配置

```yaml
# config.yaml
reranker:
  enabled: false               # 是否启用 Reranker
  provider: "llm"              # llm（用 LLM 重排序）或 api（专用 Reranker API）
  api:
    url: ""                    # Reranker API 地址
    key: ""                    # API Key
    model: ""                  # Reranker 模型
  top_k: 5                     # 重排序后返回 Top K 结果
```

Reranker 在混合搜索结果基础上做二次精排，提升搜索准确率。不启用时直接返回 RRF 融合排序结果。

### 12.4 文档管理操作

进入「知识库」页面的「文档管理」标签页：

1. **上传文档**：支持 MD / TXT / PDF / DOCX / HTML 格式，上传后自动解析和分块
2. **查看文档**：文档列表显示文件名、格式、分块数、Embedding 状态
3. **删除文档**：删除文档同时清理关联的分块和索引
4. **搜索测试**：在「搜索测试」标签页输入查询，实时查看 BM25 / 向量 / 混合搜索结果和评分
5. **重建索引**：文档内容变更后可手动触发索引重建

### 12.5 混合搜索原理

1. **BM25 搜索**：基于 TF-IDF 的关键词匹配，使用 gse 中文分词，适合精确关键词查询
2. **向量搜索**：基于 Embedding 的语义相似度，适合模糊语义查询
3. **RRF 融合**：Reciprocal Rank Fusion，将两路结果按排名倒数加权融合
4. **Reranker**（可选）：用 LLM 或专用模型对融合结果做二次精排

## 13. Runbook 增强功能

### 13.1 版本控制

Runbook 支持版本管理，每次编辑自动递增版本号：
- `version` 字段记录当前版本
- 编辑保存时 version + 1
- 可通过执行历史追溯每个版本的执行情况

### 13.2 模板变量

Runbook 步骤中支持 `{{变量名}}` 模板变量，执行时动态替换：

```markdown
## 1. 检查 CPU 使用率
ssh {{target_host}} "top -bn1 | head -20"

## 2. 检查进程
ssh {{target_host}} "ps aux --sort=-%cpu | head -{{top_n}}"
```

执行时传入参数：
```json
{
  "target_host": "192.168.1.100",
  "top_n": "10"
}
```

### 13.3 回滚步骤

Runbook 新增 `rollback_steps` 字段，定义执行失败时的回滚操作：

```markdown
## 回滚步骤
1. 恢复原配置：`cp /etc/nginx/nginx.conf.bak /etc/nginx/nginx.conf`
2. 重启服务：`systemctl restart nginx`
3. 验证恢复：`curl -s http://localhost/health`
```

### 13.4 执行历史

进入「运维手册」页面的「执行历史」标签页：
- 查看所有 Runbook 的执行记录
- 每条记录包含：执行时间、参数、状态（running/success/failed/rolled_back）、输出日志
- 支持按 Runbook 筛选和时间范围查询

### 13.5 执行 Runbook

在 Runbook 详情页点击「执行」：
1. 填写模板变量参数
2. 确认执行（auto_executable=false 的需要人工确认每一步）
3. 实时查看执行输出
4. 执行失败时可选择执行回滚步骤
