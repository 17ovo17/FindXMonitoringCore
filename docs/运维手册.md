## 3. 服务架构

```

    │
    ├── 静态页面 → Vite Dev Server (node)
    └── /api/* → Go API Server (:8080)
                    │
                    ├── MySQL (:3306) — 业务数据持久化
                    ├── Redis (:6379) — 探针在线状态缓存 + 工作流执行缓存
                    ├── Prometheus (:9090) — 监控指标查询

                    │       └── Node Exporter (:9100) — 主机指标采集

                    └── 外部 API（可选）
                            ├── Embedding API — 向量化（OpenAI 兼容）
                            └── Reranker API — 二次精排
```

## 5. 日常运维操作

### 5.1 启动全部服务

```bash
# 方式一：使用启动脚本
bash /opt/ai-workbench/start-wsl.sh

# 方式二：手动逐个启动
sudo systemctl start mysql redis-server prometheus-node-exporter
cd /opt/ai-workbench/api && nohup ./api-linux >> /opt/ai-workbench/logs/api.log 2>&1 &
prometheus --config.file=/opt/ai-workbench/docker/prometheus.yml \
  --storage.tsdb.path=/opt/ai-workbench/prometheus-data \
  --web.listen-address=0.0.0.0:9090 --web.enable-lifecycle &
cd /opt/ai-workbench/web && npm run dev -- --host 0.0.0.0 --port 3000 &
```

### 5.2 停止全部服务

```bash
bash /opt/ai-workbench/stop-wsl.sh
```

### 5.3 重启 API（代码更新后）

```bash
# 编译
cd /opt/ai-workbench/api && go build -o api-linux .

# 重启
pkill -f api-linux
nohup ./api-linux >> /opt/ai-workbench/logs/api.log 2>&1 &
```

### 5.4 查看日志

```bash
# API 日志
tail -f /opt/ai-workbench/logs/api.log

# MySQL 日志
sudo journalctl -u mysql -f

# Redis 日志
sudo journalctl -u redis-server -f
```

### 5.5 数据库备份

```bash
mysqldump -u root -p'<YOUR_PASSWORD>' -h 127.0.0.1 ai_workbench > /opt/ai-workbench/backup_$(date +%Y%m%d).sql
```

### 5.6 数据库恢复

## 9. 安全注意事项

- MySQL 已设置 root 密码（mysql_native_password 认证），Redis 当前无密码，均仅监听 127.0.0.1
- 凭据（credentials 表）中的密码当前明文存储，敏感环境需加密
- Admin Token 未配置时为 permissive 模式，生产环境必须设置 AIW_ADMIN_TOKEN
- SSH 远程执行使用 InsecureIgnoreHostKey，生产环境建议配置 known_hosts
- 远程主机白名单在 security/guard.go 中硬编码，需按实际环境修改

## 10. 原生知识库与诊断管理

### 10.1 原生能力说明

知识库、案例库、诊断工作流均由 Go API 内置实现，依赖 MySQL、Redis、Prometheus 与 LLM 配置，不需要额外 sidecar 服务。

### 10.2 知识库管理（案例库）

**进入「知识库」页面**：
- 案例分类：cpu_high / memory_leak / disk_full / oom / network_issue / connection_pool 等
- 支持关键词搜索（FULLTEXT 索引）和分类筛选
- 「新建案例」/「编辑」/「删除」单条操作
- 「导入」上传 JSON 数组批量创建
- 「导出」下载全量 JSON

**冷启动**：导入种子案例（18 条覆盖 12 个故障场景）：
```bash
curl -X POST -H 'Content-Type: application/json' \
  -d @api/assets/seed_cases.json \
  http://localhost:8080/api/v1/knowledge/cases/import
```

### 10.3 运维手册（Runbooks）

**进入「运维手册」页面**：
- 标准化故障处置流程（SOP），按分类管理
- trigger_conditions 是结构化 JSON，便于后续告警自动匹配
- steps 是 Markdown 格式，包含可执行命令
- auto_executable 标识是否允许自动执行（默认 false）

**冷启动**：导入 10 条 Runbook 种子数据：
```bash
for rb in $(jq -c '.[]' api/assets/seed_runbooks.json); do
  curl -X POST -H 'Content-Type: application/json' \
    -d "$rb" http://localhost:8080/api/v1/knowledge/runbooks
done
```

### 10.4 指标映射管理

**进入「指标映射」页面**，标准操作流程：

1. **扫描指标**：调用 Prometheus `/api/v1/label/__name__/values` 获取全量指标名（首次扫描通常 500+ 条）
2. **AI 自动适配**（开发中）：将 unmapped 指标分批发给 LLM，返回标准名 / exporter / 描述 / PromQL，写入数据库（status=auto）
3. **用户确认**：单条编辑或批量确认 AI 结果（status=auto → confirmed）
4. **本地生效**：确认后的映射写入 `metrics_mappings`，供诊断链路查询和报告生成

数据存储在 `metrics_mappings` 表，唯一约束 (datasource_id, raw_name)。

### 10.5 诊断工作流

**进入「诊断工作流」页面**：
- 输入主机名/IP + 时间范围 + 可选问题
- 响应模式：blocking 阻塞 / streaming 流式（推荐）
- 7 节点流水线：start → knowledge_retrieval → http_metrics → http_inspection → llm_diagnosis → condition → end
- 报告卡片：根因 + 证据表格 + 处置步骤 + 置信度（HIGH/MEDIUM/LOW）
- 反馈：准确 / 部分准确 / 不准确 → 写入本地反馈闭环
- 归档：将诊断结果归档为案例 → 进入本地案例库

### 10.6 反馈与归档闭环

**反馈**（POST /api/v1/diagnosis/feedback）：
- rating ∈ {accurate, partial, inaccurate}
- accurate → 记录为准确反馈，参与知识沉淀
- inaccurate → 记录为不准确反馈，进入改进样本
- partial → 记录为部分准确反馈，保留人工备注

**归档**（POST /api/v1/diagnosis/archive）：
- 将诊断记录的 raw_report 提取为 metric_snapshot
- 创建新案例，source_diagnosis_id 指向原诊断
- 归档失败时返回错误，不影响原始诊断记录

### 10.7 降级策略

| 场景 | 降级行为 |
|------|---------|
| 知识库检索为空 | 工作流跳过知识证据，继续采集指标并调用 LLM |
| Prometheus 不可用 | 指标扫描失败，工作流跳过 http_metrics 节点 |
| LLM 调用失败 | AI 自动适配返回 0 条，用户可手动编辑映射 |

### 10.8 告警驱动自动诊断

告警 webhook（POST /api/v1/alert/webhook 或 /api/v1/alert/catpaw）接收后：
1. 去重存入 alerts 表
2. 自动创建 DiagnoseRecord（trigger=alert）
3. 异步调用 RunDiagnose 进行 AI 诊断
4. 诊断完成后通过 ActionLog 关联回告警

可在「告警中心」页面查看告警关联的诊断记录。

### 10.9 诊断报告对比

GET /api/v1/diagnose/compare?ip=&limit=2 返回同主机最近 N 次诊断的 diff，便于排查问题是否复发。需要至少 2 次完成的诊断才能对比。

## 11. 原生工作流引擎

### 11.1 架构说明

原生工作流引擎所有能力编译进单一 Go 二进制，零额外运行时依赖。

```
YAML DSL 定义
    │
    ▼
dsl.go（解析 + embed.FS 加载）
    │
    ▼
graph.go（DAG 构建 + Kahn 拓扑排序 + 环检测）
    │
    ▼
engine.go（执行引擎）
    ├── 阻塞模式：同步执行，返回完整结果
    ├── 流式模式：SSE 事件推送（6 种事件类型）
    ├── 并行执行：parallel_group 节点（并行分支 + 汇聚等待）
    ├── 单节点重试：可配置重试次数和间隔
    └── 超时控制：全局 + 单节点超时
    │
    ▼
variable.go（线程安全变量池 + {{nodeID.field}} 插值）
    │
    ▼
streaming.go（SSE 事件发射器）
```

**20 种节点类型**：

| 分类 | 节点 |
|------|------|
| 基础 | start / end / condition / variable_aggregator / variable_assigner |
| LLM | llm / parameter_extractor / question_classifier |
| 数据 | knowledge_retrieval / document_extractor / http_request / tool |
| 流程 | loop / iteration / list_filter / template_transform |
| 编排 | sub_workflow / human_input |
| 代码 | code（goja JS 沙箱，13 种危险模式拦截） |
| 智能体 | agent（工具调用循环，最多 15 轮） |

**桥接层**（bridge.go）：

- `LLMBridge` — 注入 LLM 调用能力（复用 ai 配置段）
- `KnowledgeBridge` — 注入知识库检索能力（BM25 + 向量混合搜索）
- `ToolBridge` — 注入工具执行能力（HTTP 请求、Prometheus 查询等）

### 11.2 内置工作流（30 个：9 核心 + 21 别名）

**核心工作流（9 个）**：

| 工作流 | 文件 | 用途 |
|--------|------|------|
| 智能诊断 | smart_diagnosis.yaml | 知识检索 + 指标采集 + 告警采集 + LLM 根因分析 + Runbook 匹配 |
| 专项诊断 | domain_diagnosis.yaml | 根据 domain 参数选择容器、JVM、数据库、日志等专项链路 |

## 12. 知识库文档管理

### 12.1 架构说明

知识库文档管理提供完整的文档生命周期：上传 → 解析 → 分块 → Embedding → 索引 → 搜索。

```
文档上传（MD/TXT/PDF/DOCX/HTML）
    │
    ▼
parser.go（格式识别 + 内容提取）
    │
    ▼
chunker.go（智能分块）
    ├── 按段落分块（Markdown 标题感知）
    ├── 按固定长度分块（可配置 chunk_size）
    └── 重叠窗口（overlap 防止语义断裂）
    │
    ▼
indexer.go（索引构建）
    ├── BM25 索引（bm25.go，中文分词 gse）
    └── 向量索引（vector.go，外部 Embedding API）
    │
    ▼
hybrid.go（混合搜索）
    ├── BM25 关键词匹配
    ├── 向量语义相似度
    ├── RRF 融合排序
    └── Reranker 二次排序（可选）
```

### 12.2 Embedding 配置

```yaml
# config.yaml
embedding:
  provider: "builtin"          # builtin（仅 BM25）或 api（BM25 + 向量）
  api:
    url: ""                    # 向量 API 地址（OpenAI 兼容接口）
    key: ""                    # API Key（通过环境变量设置更安全）
    model: "text-embedding-3-small"  # Embedding 模型
    dimensions: 1536           # 向量维度
    batch_size: 20             # 批量请求大小
```

- `provider: "builtin"` — 仅使用 BM25 关键词搜索，无需外部 API，零成本
- `provider: "api"` — BM25 + 向量混合搜索，需要 Embedding API（支持 OpenAI、Azure OpenAI 等兼容接口）

### 12.3 Reranker 配置

```yaml
# config.yaml
reranker:
  enabled: false               # 是否启用 Reranker
  provider: "llm"              # llm（用 LLM 重排序）或 api（专用 Reranker API）
  api:
    url: ""                    # Reranker API 地址
    key: ""                    # API Key
    model: ""                  # Reranker 模型
  top_k: 5                     # 重排序后返回 Top K 结果
```

Reranker 在混合搜索结果基础上做二次精排，提升搜索准确率。不启用时直接返回 RRF 融合排序结果。

### 12.4 文档管理操作

进入「知识库」页面的「文档管理」标签页：

1. **上传文档**：支持 MD / TXT / PDF / DOCX / HTML 格式，上传后自动解析和分块
2. **查看文档**：文档列表显示文件名、格式、分块数、Embedding 状态
3. **删除文档**：删除文档同时清理关联的分块和索引
4. **搜索测试**：在「搜索测试」标签页输入查询，实时查看 BM25 / 向量 / 混合搜索结果和评分
5. **重建索引**：文档内容变更后可手动触发索引重建

### 12.5 混合搜索原理

1. **BM25 搜索**：基于 TF-IDF 的关键词匹配，使用 gse 中文分词，适合精确关键词查询
2. **向量搜索**：基于 Embedding 的语义相似度，适合模糊语义查询
3. **RRF 融合**：Reciprocal Rank Fusion，将两路结果按排名倒数加权融合
4. **Reranker**（可选）：用 LLM 或专用模型对融合结果做二次精排

## FindX Monitoring Core 运维手册

生成时间：2026-05-04 07:06（UTC+8）

本章节用于补齐 FindX Monitoring Core 的落地运维说明。当前可用能力以 `api/main.go` 中已注册的 `/api/v1/monitor/*`、`/api/v1/findx-agents/*` 为准；自动修复 `/api/v1/remediation/*` 属于规划/待实现能力，本文仅定义审批、回滚和后续验收口径，不把未实现能力描述为已完成。

### 1. 健康检查

| 检查项 | 命令或入口 | 期望结果 | 异常处理 |
|------|-----------|---------|---------|
| API 存储健康 | `curl -H "Authorization: Bearer <TOKEN>" http://127.0.0.1:8080/api/v1/health/storage` | MySQL/Redis 状态可读，错误信息不含 `<DB_DSN>` 明文 | 检查 `api/config.yaml` 占位配置、数据库进程、日志脱敏 |
| 监控核心健康 | `curl -H "Authorization: Bearer <TOKEN>" http://127.0.0.1:8080/api/v1/monitor/health` | 返回监控查询网关、默认 datasource 和基础连通状态 | 若 401/403，先确认登录态或 `X-Admin-Token: <TOKEN>`；若 503/500，检查 Query Gateway |
| 数据源健康 | `curl -H "Authorization: Bearer <TOKEN>" http://127.0.0.1:8080/api/v1/monitor/datasources` | 列出 Prometheus 类数据源，默认数据源可解析 | 检查 `monitoring.default_datasource_id`、Prometheus 地址和网络 |
| Agent 在线 | `curl -H "Authorization: Bearer <TOKEN>" http://127.0.0.1:8080/api/v1/findx-agents` | 注册 Agent 可见，心跳时间未超时 | 检查 Agent token、上报地址、目标主机时间漂移 |
| 审计链路 | `curl -H "Authorization: Bearer <TOKEN>" "http://127.0.0.1:8080/api/v1/audit/events?limit=20"` | 写操作和高风险动作有审计事件 | 若缺失，回查 handler 是否调用审计记录，避免绕过主入口 |

### 2. Target 管理

Target 是 FindX Monitoring Core 的被监控对象登记口径，当前主入口为 `/api/v1/monitor/targets`。

| 操作 | API | 权限 | 运维要点 |
|------|-----|------|---------|
| 查询列表 | `GET /api/v1/monitor/targets` | 登录态 | 支持巡检前核对目标清单，避免拓扑或告警混入无关主机 |
| 查询详情 | `GET /api/v1/monitor/targets/:id` | 登录态 | 用于排障单点目标的 labels、datasource、启停状态 |
| 新增/更新 | `POST /api/v1/monitor/targets`、`PUT /api/v1/monitor/targets/:id` | 管理员 | 提交前确认目标唯一标识、业务归属、数据源映射；敏感字段使用 `<TOKEN>`、`<SECRET>` 占位 |
| 删除 | `DELETE /api/v1/monitor/targets/:id` | 管理员 | 删除前确认没有活跃告警、自动修复计划或未归档事件引用 |

排障建议：
- Target 查不到指标时，先用 `/api/v1/monitor/label-values` 查询 `instance`、`job` 等标签，再确认 target labels 是否与 PromQL 匹配。
- 大批量导入前先抽样 3-5 个目标做 query 验证，避免大基数标签导致候选截断或 UI 列表卡顿。
- 删除或改名目标需要记录审计备注，便于后续事件回溯。

### 3. Datasource 管理

当前 FindX Monitoring Core 的 datasource 以 Prometheus 兼容查询为核心，主入口为 `/api/v1/monitor/datasources` 和 Query Gateway。

| 场景 | 操作 |
|------|------|
| 新接入 Prometheus | 在配置或管理界面登记 datasource id、base URL、超时和默认状态；任何认证信息写为 `<TOKEN>` 或通过环境变量注入 |
| 切换默认数据源 | 调整 `monitoring.default_datasource_id` 后重启 API，并执行 `/api/v1/monitor/health`、`/api/v1/monitor/query` 烟测 |
| 数据源不可达 | 先验证 Prometheus `/-/ready`，再验证 API 到 Prometheus 的网络；必要时启用只读降级，不创建新告警事件 |
| 多数据源混用 | Alert Rule、Target 和 Query Gateway 请求必须显式携带 datasource id，避免跨环境串查 |

禁止事项：
- 不在文档、日志、测试报告中写完整 Prometheus Basic Auth、Bearer Token 或 `<DB_DSN>` 明文。
- 不把测试 datasource 指向生产环境，除非有隔离的只读账号和明确测试窗口。

### 4. Query Gateway 排障

Query Gateway 当前通过 `/api/v1/monitor/query`、`/api/v1/monitor/query-range`、`/api/v1/monitor/metrics`、`/api/v1/monitor/labels`、`/api/v1/monitor/label-values` 对外提供 Prometheus 兼容查询能力。

| 问题 | 快速判断 | 处理方式 |
|------|---------|---------|
| 401/403 | 请求缺少 `Authorization: Bearer <TOKEN>` 或管理员 token | 重新登录或使用占位 token 对应的测试账号；不要在命令历史中写真实 token |
| 400 | PromQL、时间范围、step、label 参数不合法 | 先用最小 PromQL `up` 验证，再逐步加入标签；Workflow API 400 风险需同步测试矩阵回归 |
| 503/500 | Prometheus 不可达、超时或响应结构异常 | 检查 datasource URL、Prometheus ready、API 日志；响应和日志必须脱敏 |
| 空结果 | 标签不匹配或时间窗口无数据 | 用 `/labels`、`/label-values` 找真实 label，再缩短时间窗口或切换 `query_range` |
| 大基数慢查询 | 返回候选过多或被截断 | 增加 label 过滤，控制 `limit`，记录“候选截断”风险，不把不完整候选作为最终诊断证据 |

标准排障命令示例：

```bash
curl -H "Authorization: Bearer <TOKEN>" \
  -H "Content-Type: application/json" \
  -d '{"datasource_id":"default","query":"up"}' \
  http://127.0.0.1:8080/api/v1/monitor/query
```

### 5. Alert Rule 管理

主入口为 `/api/v1/monitor/alert-rules`。当前代码已注册规则 CRUD、启停、克隆、试运行和回滚入口。

| 操作 | API | 验收点 |
|------|-----|-------|
| 查询规则 | `GET /api/v1/monitor/alert-rules` | 能看到 enabled、status、severity、datasource、query |
| 创建规则 | `POST /api/v1/monitor/alert-rules` | PromQL 校验通过，默认不泄露 token/URL 凭据 |
| 更新规则 | `PUT /api/v1/monitor/alert-rules/:id` | 版本记录可追踪，旧版本可回滚 |
| 启用/禁用 | `POST /api/v1/monitor/alert-rules/:id/enable`、`/disable` | 调度器只评估 enabled 且 active 的规则 |
| 克隆 | `POST /api/v1/monitor/alert-rules/:id/clone` | 新规则 ID 独立，默认不直接影响生产告警 |
| 试运行 | `POST /api/v1/monitor/alert-rules/:id/tryrun` | 只返回评估结果，不创建正式事件 |
| 回滚 | `POST /api/v1/monitor/alert-rules/:id/rollback` | 回到指定版本后需再次 tryrun |

变更流程：
1. 创建或修改规则前记录需求来源、目标 datasource、PromQL、阈值和预期候选数量。
2. 先 `tryrun`，确认正常、异常、空结果、大基数边界。
3. 再启用规则，观察至少一个调度周期。
4. 若误报或查询异常，优先 disable，再 rollback，不直接删除证据。

### 6. Event 处置

主入口为 `/api/v1/monitor/events/current`、`/api/v1/monitor/events/history` 和事件动作 API。

| 状态动作 | API | 处置要求 |
|---------|-----|---------|
| 查看当前事件 | `GET /api/v1/monitor/events/current` | 先按 severity、rule_id、target_ident 聚合，识别重复事件 |
| 查看历史事件 | `GET /api/v1/monitor/events/history` | 用于复盘和误报追踪 |
| 查看详情 | `GET /api/v1/monitor/events/:id` | 核对 labels、annotations、query hash、证据来源 |
| 确认 | `POST /api/v1/monitor/events/:id/ack` | 记录确认人、原因和处理窗口 |
| 分派 | `POST /api/v1/monitor/events/:id/assign` | 指定 owner，不写真实个人敏感联系信息 |
| 解决 | `POST /api/v1/monitor/events/:id/resolve` | 必须填写原因和恢复证据 |
| 归档 | `POST /api/v1/monitor/events/:id/archive` | 仅对已解决或误报事件归档 |

事件闭环最小证据：
- 触发规则、PromQL、datasource id、target ident。
- Query Gateway 返回摘要，敏感 label 已脱敏。
- AI 问诊证据链或人工排障记录。
- 若涉及自动修复，必须包含审批、执行结果和回滚入口。

### 7. Evaluator/Scheduler 排障

当前 `scheduler.StartMonitorAlertScheduler()` 在 API 启动时注册监控告警调度器，配置项以 `monitoring.alert_scheduler.*` 为主。

| 排障点 | 观察方式 | 处理建议 |
|------|---------|---------|
| 调度器未启动 | 查看 API 日志中 `monitor alert scheduler` 相关记录 | 检查 `monitoring.alert_scheduler.enabled`；禁用状态只允许手工 tryrun |
| 并发跳过 | 日志出现 skipped 或上次运行未结束 | 降低规则数量、缩短单次 query 超时、检查慢 PromQL |
| 规则评估失败 | 查看 eval log、规则详情、Query Gateway 响应 | 区分 PromQL 400、datasource 503、Evaluator 阈值错误 |
| 事件未创建 | 核对候选为空、规则状态、事件 fingerprint | 检查大基数候选截断和 fingerprint 去重 |
| 持久化失败 | API 日志有 DB 写入错误或内存 fallback | 按 P1 风险处理：确认 current/history/action log 一致性，恢复后补做回归 |

降级原则：
- Query Gateway 不可用时，Scheduler 不应创建缺证据的新 firing 事件。
- Evaluator 返回 no_data 时按规则策略处理，不能编造候选。
- 持久化失败时，API 可返回错误或进入内存 fallback，但测试报告必须标明一致性风险。

### 8. findx-agents 安装、升级与回滚

当前 FindX Agent 的服务端主入口为：
- `GET /api/v1/findx-agents`
- `POST /api/v1/findx-agents/register`
- `POST /api/v1/findx-agents/heartbeat`

安装流程：
1. 在目标主机创建独立运行用户，准备配置目录。
2. 写入 Agent 配置，示例仅允许占位符：

```toml
server_url = "http://<AI_WORKBENCH_HOST>:8080"
agent_token = "<TOKEN>"
agent_id = "<AGENT_ID>"
target_id = "<TARGET_ID>"
heartbeat_interval_seconds = 30
```

3. 启动后调用注册接口，确认 `/api/v1/findx-agents` 可查询到 Agent。
4. 连续观察 2-3 个心跳周期，确认状态、版本和 target 归属。

升级流程：
1. 记录当前二进制版本、配置 checksum、运行用户和启动方式。
2. 停止 Agent 或进入维护模式。
3. 替换二进制，保留原二进制为 `findx-agent.bak.<version>`。
4. 启动新版本，验证 register/heartbeat、日志脱敏、CPU/内存占用。

回滚流程：
1. 停止新版本 Agent。
2. 恢复备份二进制和原配置。
3. 启动旧版本，确认 heartbeat 恢复。
4. 在审计记录或变更单中记录回滚原因、影响范围和后续修复计划。

### 9. AI 问诊证据链

AI 问诊不得直接替代监控证据。一次可验收的问诊链路至少包含：
- 用户问题、目标对象、时间范围。
- Query Gateway 查询摘要：PromQL、datasource id、时间窗口、结果统计。
- Alert Event 证据：event id、rule id、severity、状态流转。
- Target/Agent 证据：target id、agent id、最后心跳。
- 知识库或 Runbook 引用：文档 id、命中片段摘要。
- AI 输出：根因假设、置信度、证据引用、未覆盖数据缺口。

禁止项：
- Prometheus/Catpaw/FindX Agent 无数据时，AI 不得编造指标。
- AI 输出中不得包含真实 token、Cookie、完整 `<DB_DSN>`、SSH 私钥或会话 ID。
- 自动修复建议在 `/api/v1/remediation/*` 未实现前只能标为“建议/待人工执行/后续验收”。

### 10. 自动修复审批与回滚

当前 `/api/v1/remediation/*` 尚未在 API 路由中注册，属于规划/待实现。运维落地先采用人工审批和 Runbook 执行记录。

规划中的自动修复门禁：
- `POST /api/v1/remediation/plans`：根据事件生成修复计划，待实现。
- `POST /api/v1/remediation/plans/:id/approve`：审批计划，待实现。
- `POST /api/v1/remediation/plans/:id/execute`：执行计划，待实现。
- `POST /api/v1/remediation/plans/:id/rollback`：按回滚步骤恢复，待实现。
- `GET /api/v1/remediation/plans/:id`：查看计划、审批、执行和回滚证据，待实现。

上线前必须验收：
- L0/L1/L2/L3/L4 命令分级和拦截。
- 双人审批或管理员二次确认。
- 幂等执行、重复提交保护、失败中止。
- 回滚步骤可独立执行，且回滚前后有状态快照。
- 审计事件包含 actor、target、risk、decision、trace id，敏感字段脱敏。

### 11. 降级策略

| 故障场景 | 降级行为 | 恢复动作 |
|---------|---------|---------|
| MySQL 不可用 | 只读健康检查返回失败；部分模块可能进入内存 fallback，但不得声明持久化成功 | 恢复 MySQL 后对比 current/history/action log 一致性 |
| Redis 不可用 | 缓存和短期状态降级；核心监控查询不应依赖 Redis 才可读 | 恢复 Redis 后观察缓存命中和调度状态 |
| Prometheus 不可用 | Query Gateway 返回明确错误；Alert Scheduler 不创建缺证据的新事件 | 恢复 Prometheus 后先执行 `up` 和关键规则 tryrun |
| Agent 离线 | Target 仍可查询，但 Agent 证据标记缺失或过期 | 检查 Agent 进程、网络、token、时间同步 |
| AI Provider 不可用 | AI 问诊降级为规则/Runbook/监控证据摘要 | 恢复后补做问诊，不覆盖人工处置记录 |
| Remediation 未实现 | 所有自动修复只标记为规划/待实现，不执行线上变更 | 待接口实现后按专项矩阵重新验收 |

### 12. 审计查询

审计查询主入口为 `GET /api/v1/audit/events`。建议按以下问题追踪：

| 查询目标 | 关注字段 |
|---------|---------|
| 规则变更 | action、target、operator、decision、integrity_hash |
| 事件处置 | event id、ack/assign/resolve/archive 动作、reason |
| Agent 生命周期 | register、heartbeat 异常、升级/回滚记录 |
| 自动修复审批 | remediation plan id、approver、risk、rollback status（待实现） |
| 敏感信息风险 | 日志和审计中不得出现真实 `<TOKEN>`、Cookie、完整 `<DB_DSN>`、SSH 私钥 |

审计留存建议：
- 线上保留最近 90 天审计记录，归档前做脱敏检查。
- 对 P0/P1 事件保留完整证据链，包含 query 摘要、event action log、AI 问诊摘要和回滚记录。
- 若发现 integrity hash 缺失或不一致，按 P1 风险处理，暂停自动修复和高风险写操作。
