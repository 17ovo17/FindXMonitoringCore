## 1. 规划目标与边界

FindX 是面向运维、平台工程、业务稳定性团队的全栈可观测与 AI SRE 平台。长期目标不是把多个监控页面简单拼接，也不是只提供 AI 问答入口，而是形成以 **FindX Core** 为平台控制面、以 **findx-agents** 为主机与采集执行控制面、以 **Evidence Chain** 为证据骨架、以 **AI SRE** 为诊断与处置入口、以 **Remediation** 为修复闭环的 Agent-First 可观测体系。

本规划只定义长期方向、产品语义、能力边界、架构关系和验收口径。日志、APM、RUM、on-call、Status Page 等能力均属于长期能力，不在本轮文档中表达为已完成能力。任何后续实现仍需由独立任务、代码评审、构建验证、接口验证和 QA 回归承接。

命名边界必须保持稳定：

- 产品主语统一使用 FindX、FindX Core、findx-agents、AI SRE、Evidence Chain、Remediation。
- 用户侧导航、页面标题、任务名称、接口分组、权限对象和审计对象统一使用 FindX 语义。
- 成熟开源监控平台设计可作为内部参考，但不得转化为 FindX 的模块名、菜单名、API 名或数据对象名。
- 通知与响应协同使用 FindX 用户、团队、角色、业务空间、排班、升级策略，不创造额外的独立人员组织对象。
- 健康检查按对象归属 AI SRE 或资产与采集联动，平台治理只保留平台运行自检。

Agent-First 闭环的核心链路如下：

1. FindX Core 下发策略、配置、权限和任务。
2. findx-agents 采集指标、日志、Trace、事件、巡检结果和结构化命令输出。
3. Evidence Chain 把告警、查询、巡检、诊断、修复和审计串联为可验证证据。
4. AI SRE 基于证据给出诊断、建议、修复计划和复盘结论。
5. Remediation 在审批、安全策略、回滚条件和审计约束下执行修复动作。

## 2. FindX 平台理念

FindX 的长期理念是“全栈信号统一、证据驱动诊断、Agent-First 执行、安全可审计修复”。平台不是单点工具集合，而是从采集、存储、查询、告警、响应、诊断、修复到复盘的闭环系统。

FindX Core 是平台控制面和统一工作台。它长期承载资产管理、采集接入、数据查询、指标分析、日志分析、Trace 分析、RUM 分析、告警响应、响应协同、AI SRE 编排、权限治理、审计追踪和平台运行自检。FindX Core 不直接承接主机健康检查的产品语义，主机和服务运行状态检查由 AI SRE 与资产、采集、findx-agents 联动完成。

findx-agents 是 FindX 面向主机、服务、运行环境和采集插件的核心控制面，不是边缘附庸。它负责统一采集器发行版、注册心跳、能力目录、Categraf 插件目录、采集配置中心化下发、配置版本 diff 和回滚、远程安装、本机安装指令、凭证引用、远程调试、自升级、结构化命令执行、告警自愈、Catpaw 衍生单机巡检、诊断会话、证据产物和审计脱敏。

Evidence Chain 是 FindX 对 AI SRE 可信度的要求。任何诊断结论、告警归因、修复建议和自动化执行都必须能回到可验证证据。证据可以来自指标、日志、Trace、事件、拓扑、配置快照、巡检结果、命令输出、变更记录和审计记录。没有证据链支撑的 AI 结论只能作为提示，不能直接触发 Remediation。

AI SRE 是证据驱动的 SRE 协作者，长期负责主机健康检查、AI 健康诊断、单机巡检、Agent 巡检、服务健康检查、诊断会话、根因分析、修复建议、修复编排和复盘沉淀。AI SRE 不应被定义为普通问答模块，而应被定义为围绕 Evidence Chain 与 Remediation 运行的诊断处置入口。

Remediation 是 FindX 的修复闭环能力。长期目标包括手动确认修复、半自动修复、低风险自动执行、失败回滚、变更前检查、变更后验证、执行审计、失败降级和复盘记录。所有修复动作必须可审批、可回滚、可审计、可复核。

## 7. 功能去重归并表

长期规划需要避免同一能力在多个入口重复出现，形成并行定义或冲突命名。去重归并原则如下：

| 能力 | 归并后入口 | 不再扩散到 | 说明 |
|---|---|---|---|
| 主机健康检查 | AI SRE / 资产与采集联动 | 平台治理 | 平台治理只保留平台运行自检 |
| 单机巡检 | AI SRE / findx-agents inspector | 独立产品入口 | Catpaw 衍生能力进入 findx-agents 内部落点 |
| Agent 巡检 | AI SRE / 资产与采集 | 平台治理 | 关注心跳、版本、插件、配置和任务执行状态 |
| 服务健康检查 | AI SRE | 平台治理 | 面向服务端口、进程、依赖、错误率和资源瓶颈 |
| AI 健康诊断 | AI SRE | 普通问答入口 | 必须基于 Evidence Chain |
| 平台运行自检 | 平台治理 | AI SRE 主机检查 | 只检查 FindX 平台自身组件、依赖、任务和配置 |
| 插件目录 | Agent 管理中心 / 资产与采集 | 独立外部产品入口 | Categraf 作为采集插件底座，由 findx-agents 纳管 |
| 配置版本 diff / 回滚 | Agent 管理中心 | 分散配置页 | 统一处理采集配置、巡检配置和插件配置 |
| 诊断会话 | AI SRE | 普通工具执行页 | 会话必须关联证据产物和修复计划 |
| Remediation | AI SRE / 工作流 | 任意命令入口 | 修复动作必须审批、审计、回滚、验证 |
| 通知、排班、升级策略 | 响应协同 | 告警规则内部重复建模 | 告警只引用响应协同策略 |

去重后的核心判断是：执行面归 findx-agents，诊断面归 AI SRE，证据归 Evidence Chain，修复归 Remediation，平台自身状态归平台运行自检。

## 8. findx-agents 理念

findx-agents 是 FindX 的核心控制面之一，不是单个采集进程，也不是边缘附庸。它长期承担“采集插件底座 + 主机接入控制协议 + 配置中心 + 巡检诊断 + 结构化工具 + Remediation 执行 + 安全审计”的统一执行层。

findx-agents 必须覆盖以下能力：

| 能力 | 长期要求 |
|---|---|
| 统一采集器发行版 | 以统一发行版承载主机接入、采集插件、巡检诊断、远程调试、自升级和安全执行 |
| 注册心跳 | Agent 向 FindX Core 注册身份、版本、业务空间、标签、运行状态和心跳 |
| 能力目录 | 上报采集插件、巡检能力、诊断工具、执行能力和权限要求 |
| Categraf 插件目录 | 统一展示插件能力、版本、配置模板、适用场景、依赖和状态 |
| 采集配置中心化下发 | 支持按主机、标签、业务空间、团队授权批量下发 |
| 配置版本 diff / 回滚 | 对比配置版本、插件版本和目标主机实际状态，支持灰度和回滚 |
| 远程安装 | 通过凭证引用和授权任务完成受控安装，输出脱敏审计 |
| 本机安装指令 | 提供可复制的最小安装指令，令牌示例只能使用 `<TOKEN>` |
| 凭证引用 | 只引用 `<SSH_KEY_REF>` 等凭证对象，不在页面、日志或报告中展示真实凭证 |
| 远程调试 | 支持受控调试、超时、输出脱敏、证据归档和审计留痕 |
| 自升级 | 支持版本检查、灰度升级、失败降级、回滚和升级后验证 |
| 结构化命令执行 | 工具输入输出必须结构化，带权限、风险、超时和脱敏规则 |
| 告警自愈 | 低风险场景可由告警触发 Remediation，但必须有证据、审批策略和回滚 |
| Catpaw 衍生单机巡检 | 单机巡检思想进入 inspector，不形成用户侧独立产品入口 |
| 诊断会话 | diagnosis session 关联告警、主机、证据、工具输出和修复计划 |
| 证据产物 | evidence artifact 可检索、可聚合、可追踪、可复盘 |
| 审计脱敏 | 所有任务、命令、配置、巡检、修复和调试输出必须脱敏审计 |

findx-agents 与 FindX Core 的关系应保持“控制面协同”：FindX Core 管策略、权限、审计和全局编排，findx-agents 管主机执行、插件运行、巡检诊断和本地证据采集。任何绕过权限模型的任意命令执行都不应进入长期规划。

## 9. Agent 管理中心设计

Agent 管理中心是 findx-agents 能力的用户侧管理入口，长期用于统一管理主机接入、Agent 状态、插件目录、配置下发、远程调试、自升级、巡检诊断和审计记录。本章是规划设计，不代表当前页面已经实现。

建议信息架构如下：

| 子域 | 核心能力 | 验收边界 |
|---|---|---|
| Agent 总览 | 在线数、离线数、版本分布、插件状态、异常趋势 | 不展示敏感凭证或完整连接串 |
| 接入向导 | 本机安装指令、远程安装、批量接入、业务空间绑定 | 示例使用 `<TOKEN>`、`<LOGIN_USER>`、`<SSH_KEY_REF>` |
| 注册心跳 | 注册状态、心跳时间、版本、标签、能力上报 | 支持异常诊断和重新注册说明 |
| 能力目录 | 采集能力、巡检能力、工具能力、执行权限 | 能力必须可审计、可授权、可禁用 |
| 插件目录 | Categraf 插件目录、配置模板、默认参数、适用场景 | 插件生态纳入 findx-agents，不独立成外部入口 |
| 配置中心 | 中心化下发、版本 diff、灰度发布、reload、回滚 | 每次变更产生配置版本和审计记录 |
| 远程调试 | 受控调试、输出脱敏、证据归档、超时控制 | 不允许无权限命令执行 |
| 自升级 | 版本检查、批次策略、升级前检查、升级后验证 | 支持失败降级和回滚 |
| 巡检诊断 | 主机巡检、Agent 巡检、服务检查、结构化工具 | 结果进入 Evidence Chain 并供 AI SRE 引用 |
| 审计记录 | 任务下发、命令执行、配置变更、修复动作、调试记录 | 审计必须脱敏并关联操作者、目标和证据 |

Agent 管理中心不应把 findx-agents 降级为“采集器列表”。它应表达为 FindX 的主机执行与采集控制面，支撑 AI SRE 和 Remediation 的可信闭环。

## 10. Catpaw 衍生融合边界

Catpaw 的价值在于单机巡检、诊断工具、会话化执行、结构化输出、工具注册和安全执行思想。FindX 可以吸收这些思想，但用户侧产品入口、导航、接口分组和长期品牌必须统一使用 FindX / findx-agents。

能力映射如下：

| Catpaw 思想 | findx-agents 内部落点 | FindX 用户侧表达 |
|---|---|---|
| 单机巡检 | inspector | AI SRE / 主机健康检查 |
| 诊断工具 | tool registry | 工具目录 / 诊断工具 |
| 会话化执行 | diagnosis session | AI 诊断会话 |
| 结构化输出 | evidence artifact | 证据产物 / Evidence Chain |
| 工具注册 | tool registry | 工具目录 |
| 安全执行思想 | remediation executor | Remediation / 安全执行 |

融合边界必须清楚：

- Catpaw 只作为内部能力来源和设计吸收对象，不作为用户侧主产品入口。
- inspector、diagnosis session、tool registry、evidence artifact、remediation executor 是 findx-agents 内部能力落点。
- 单机巡检和诊断工具必须区分只读检查、低风险操作和高风险操作。
- 高风险操作必须具备审批、确认、超时、最小权限、输出脱敏、回滚策略和审计记录。
- 远程执行不能绕过 FindX Core 的权限模型，也不能绕过业务空间授权。
- 证据产物必须关联诊断会话、告警事件、目标主机、执行人、执行时间和脱敏输出。

## 11. Categraf 结合方式

Categraf 作为采集插件底座纳入 findx-agents，不作为独立用户侧产品入口。FindX 的长期重点不是重写插件生态，而是把插件目录、配置模板、中心化下发、版本 diff、reload、回滚、自升级和远程调试纳入 findx-agents 控制面。

结合方式如下：

| 能力 | 规划要求 |
|---|---|
| 插件目录 | 展示插件名称、类型、版本、适用场景、依赖、默认参数和状态 |
| 配置模板 | 管理模板变量、默认值、校验规则、业务空间覆盖和敏感字段引用 |
| 中心化下发 | 支持按主机、标签、业务空间、团队授权批量下发采集配置 |
| 版本 diff | 对比模板版本、目标配置、运行配置和历史版本 |
| reload | 支持安全 reload、失败降级、结果回执和证据归档 |
| 回滚 | 支持配置回滚、插件版本回滚、灰度撤销和回滚审计 |
| 自升级 | 支持 Agent 与插件版本检查、灰度升级、失败降级和升级验证 |
| 远程调试 | 支持插件连通性检查、采集样本检查、输出脱敏和审计留痕 |

长期验收不以“插件能运行”为唯一标准，而以“插件可发现、配置可治理、变更可回滚、异常可诊断、执行可审计”为标准。

## 12. 指标平台规划

指标平台是 FindX 全栈可观测的基础能力，应优先形成稳定的数据模型、查询体验、模板体系和告警联动。指标能力可以先于日志、APM 和 RUM 成熟，但不能孤立存在，必须能进入 Evidence Chain。

长期规划能力：

- 指标接入：主机、服务、数据库、中间件、业务指标和自定义指标。
- 指标查询：支持聚合、筛选、分组、时间对比、查询收藏和查询历史。
- 指标质量：关注采集延迟、缺失、异常值、标签基数和采集错误。
- 指标模板：按主机、服务、业务空间和插件生成仪表盘模板。
- 容量趋势：支持 CPU、内存、磁盘、网络、服务吞吐和自定义容量指标趋势。
- 告警联动：指标查询可被告警规则引用，告警事件可反查指标证据。
- AI SRE 联动：指标异常可进入诊断会话，成为根因分析和 Remediation 的证据。

边界要求：

- 指标平台不承接主机健康检查的归属，健康检查由 AI SRE 或资产与采集联动完成。
- 指标平台可以提供指标证据，但诊断结论必须由 AI SRE 基于 Evidence Chain 形成。
- 指标查询与展示不使用外部产品命名作为用户侧模块名。

## 13. 日志平台规划

日志平台是长期能力，本轮规划不表达为已完成。日志能力的目标是为故障定位、审计追踪、AI SRE 诊断和 Remediation 验证提供文本与结构化事件证据。

长期规划能力：

- 日志接入：支持主机日志、应用日志、容器日志、系统日志和业务日志。
- 解析规范：支持字段解析、时间解析、级别识别、服务标识、业务空间标识和 Trace 关联字段。
- 检索分析：支持关键词、字段过滤、时间窗口、上下文展开、聚合统计和分析视图。
- 脱敏治理：对 Token、Cookie、DSN、密钥、账号、连接串等敏感信息进行采集侧和展示侧脱敏。
- 留存策略：按业务空间、日志类型、重要级别和合规要求配置保留周期。
- 告警联动：日志模式、错误频率、异常关键字和结构化字段可触发告警事件。
- AI SRE 联动：日志片段进入 Evidence Chain，供诊断会话引用。

验收边界：

- 示例只能使用 `<TOKEN>`、`<API_KEY>`、`<DB_DSN>`、`<LOGIN_USER>` 等占位符。
- 日志平台不能展示堆栈中的敏感配置、完整连接串或认证票据。
- 日志能力不得替代审计日志；审计日志由平台治理和执行审计负责。

## 14. APM / Trace 平台规划

APM / Trace 平台是长期能力，本轮规划不表达为已完成。该能力用于连接服务拓扑、调用链、慢调用、错误聚合、依赖分析和业务影响面。

长期规划能力：

- Trace 接入：支持服务、接口、任务、消息队列、数据库调用和外部依赖调用。
- 服务拓扑：基于 Trace、指标和配置关系生成服务依赖视图。
- 慢调用分析：支持按服务、接口、依赖、时间窗口和业务空间聚合。
- 错误聚合：把异常、状态码、调用失败、超时和依赖错误聚合为可诊断事件。
- 跨信号关联：Trace 与指标、日志、告警、变更和审计记录互相跳转。
- AI SRE 联动：Trace 片段进入诊断会话，辅助根因分析和修复计划。
- Remediation 验证：修复后通过 Trace 指标验证错误率、耗时和依赖状态是否恢复。

边界要求：

- APM / Trace 平台提供证据，不直接替代 AI SRE 的诊断结论。
- Trace 数据必须遵守业务空间权限和敏感字段脱敏。
- 服务拓扑不应暴露内部密钥、完整连接串或未授权依赖详情。

## 15. RUM 平台规划

RUM 是长期能力，本轮规划不表达为已完成。RUM 的目标是从用户体验视角补齐可观测信号，使 FindX 能把前端体验、后端依赖、告警事件和 AI SRE 诊断连接起来。

长期规划能力：

- 页面性能：采集加载耗时、资源耗时、交互延迟、接口耗时和页面可用性。
- 前端错误：采集脚本错误、资源加载失败、接口失败、路由错误和异常上下文。
- 用户体验路径：按业务空间、页面、功能路径和时间窗口观察体验质量。
- 会话关联：在隐私保护和授权范围内关联用户体验、Trace、日志和告警事件。
- 告警联动：页面错误率、接口失败率、核心路径耗时可触发告警。
- AI SRE 联动：RUM 信号进入 Evidence Chain，辅助判断业务影响面。

隐私与安全要求：

- 不采集明文密码、认证票据、Cookie、Token、完整连接串、身份证件或支付信息。
- 采样、脱敏、留存和访问权限必须按业务空间和角色控制。
- RUM 结论必须标注采样范围和证据来源，避免把局部体验误判为全局故障。

## 16. 告警与事件响应规划

告警与事件响应负责把可观测信号转化为可处置事件。长期目标是从规则触发走向事件收敛、证据聚合、响应协同、AI SRE 诊断和 Remediation 闭环。

长期规划能力：

- 告警规则：支持指标、日志、Trace、RUM、平台运行自检和自定义信号。
- 事件中心：对规则触发结果进行收敛、去重、关联、影响面判断和状态流转。
- 静默与订阅：支持按业务空间、服务、级别、标签、时间窗口进行控制。
- 处理流水线：支持分派、确认、处理中、观察、恢复、关闭和复盘状态。
- 证据聚合：告警事件必须关联触发数据、相关日志、Trace、巡检结果和审计记录。
- AI SRE 入口：严重或复杂事件可创建诊断会话。
- Remediation 入口：满足条件的低风险事件可触发受控修复计划。

安全边界：

- 告警不能直接触发无审批的高风险 Remediation。
- 告警通知中不得展示真实密钥、Cookie、Token、DSN、SSH 私钥或完整连接串。
- 告警事件应记录规则版本、触发条件、证据快照、处理人和状态变化。

## 17. On-call、通知与 Status Page 规划

On-call、通知与 Status Page 均属于长期能力，本轮规划不表达为已完成。它们应统一归入响应协同，而不是分散在告警规则或人员组织对象中。

长期对象模型：

- FindX 用户：通知接收人、处理人、审批人和审计主体。
- 团队：协作组织，可绑定业务空间、服务范围和响应责任。
- 角色：权限集合，控制查看、配置、审批、执行和审计能力。
- 业务空间：稳定性治理边界，用于组织资产、告警、团队和响应策略。
- 排班：定义时间段内的响应责任人。
- 升级策略：定义未响应、超时、严重级别变化后的升级路径。
- 通知渠道：承载消息发送通道、模板、重试、抑制和发送记录。
- Status Page：长期用于对外或对内展示服务状态、事件进展和恢复记录。

规划边界：

- 通知能力不创建额外独立人员组织对象，统一复用 FindX 用户、团队、角色、业务空间、排班、升级策略。
- Status Page 不能泄露内部拓扑、敏感日志、凭证引用、完整连接串或未公开事件细节。
- on-call 相关策略必须可审计，并与告警事件、处理流水线和复盘记录关联。

## 18. 仪表盘与模板体系规划

仪表盘与模板体系负责把指标、日志、Trace、RUM、告警、巡检和 AI SRE 结论组织成可复用视图。长期目标是建立“模板复用、证据可跳转、业务空间隔离、变更可审计”的展示体系。

长期规划能力：

- 仪表盘：支持按主机、服务、业务空间、团队、告警事件和诊断会话组织视图。
- 图表：支持指标趋势、日志聚合、Trace 耗时、RUM 性能、告警状态和巡检结果。
- 模板：支持主机模板、服务模板、插件模板、业务空间模板和事件复盘模板。
- 事件标记：支持发布、变更、告警、Remediation 执行和恢复节点的证据时间线标记。
- 对比分析：支持时间对比、实例对比、版本对比和业务空间对比。
- 证据跳转：图表应能跳转到原始指标、日志、Trace、告警事件和诊断会话。
- 权限隔离：仪表盘、模板和数据源必须遵守业务空间与角色权限。

命名边界：

- 用户侧统一使用“仪表盘与模板”表达，不使用独立外部命名。
- 仪表盘模板不替代 Evidence Chain；它是证据展示入口，不是诊断结论本身。
- 模板复用必须避免跨业务空间泄露敏感数据。

## 19. AI SRE、健康检查、知识库、工作流与自动修复规划

AI SRE 是 FindX 长期差异化能力的核心。它应把健康检查、诊断会话、知识库、工作流、Evidence Chain 和 Remediation 串成闭环。

健康检查归属必须保持如下边界：

| 检查类型 | 归属 | 说明 |
|---|---|---|
| 主机健康检查 | AI SRE / 资产与采集联动 | 面向主机资源、系统状态、服务状态和运行风险 |
| 单机巡检 | AI SRE / findx-agents | 由 findx-agents inspector 执行，结果进入诊断会话 |
| Agent 巡检 | AI SRE / 资产与采集联动 | 关注 Agent 心跳、版本、插件、配置和任务执行状态 |
| 服务健康检查 | AI SRE | 面向服务进程、端口、依赖、错误率和资源瓶颈 |
| AI 健康诊断 | AI SRE | 基于 Evidence Chain 形成诊断建议和复盘 |
| 平台运行自检 | 平台治理 | 只检查 FindX 平台自身组件、依赖、任务和配置状态 |

AI SRE 长期能力：

- 诊断会话：围绕告警、主机、服务、业务空间或巡检任务组织证据。
- Evidence Chain：关联指标、日志、Trace、RUM、事件、巡检、命令输出、配置快照和审计记录。
- 知识库：沉淀 Runbook、故障模式、指标解释、处置策略和复盘结论。
- 工作流：编排检查步骤、诊断步骤、审批步骤、修复步骤和验证步骤。
- Remediation：执行前预检、计划预览、人工确认、低风险自动执行、高风险审批、失败回滚、执行后验证。
- 告警自愈：由告警触发受控诊断和低风险修复，但必须满足证据充分、权限允许、策略批准、回滚可用。
- 复盘沉淀：把诊断结论、证据、修复动作、验证结果和遗留风险沉淀回知识库。

平台治理只保留平台运行自检。平台运行自检可以检查 FindX Core、任务调度、存储依赖、模型配置、消息通道、审计链路和内部配置状态，但不承载主机健康检查、单机巡检、Agent 巡检、服务健康检查或 AI 健康诊断。

## 20. 后端架构、数据存储与 API 契约规划

本章只描述长期架构规划，不实施业务代码，不新增数据表，不修改接口契约。后续任何 API_CONTRACT_CHANGE 或 DATA_CHANGE 都必须在独立任务中明确旧契约、新契约、兼容策略、迁移方案、回滚方式和验证命令。

长期后端分层建议：

| 层级 | 职责 | 边界 |
|---|---|---|
| Handler | 接收请求、鉴权、参数校验、错误响应 | 不直接访问存储细节，不拼接敏感信息 |
| Service | 业务编排、权限上下文、流程控制、降级策略 | 不绕过领域模型，不吞错误 |
| Store | 数据读写、事务、索引、幂等和兼容处理 | 不承载业务决策 |
| Model | 状态、枚举、领域对象、响应结构 | 状态值集中治理 |
| Security | 权限、凭证引用、脱敏、审计、高危确认 | 不暴露真实密钥或完整连接串 |
| Agent Control | Agent 注册、心跳、能力目录、配置下发、任务回执 | 与 findx-agents 协议稳定兼容 |
| Evidence | 证据产物、证据链索引、诊断会话关联 | 支持查询、审计和复盘 |
| Remediation | 修复计划、审批、执行、验证、回滚 | 严格依赖权限、证据和审计 |

数据与 API 契约长期要求：

- API 路径、方法、请求字段、响应字段、错误码和认证要求发生变化时必须打标 API_CONTRACT_CHANGE。
