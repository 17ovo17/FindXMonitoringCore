# Claude 不在时的主代理接管协作模式

## 背景

本讨论文档记录 FindX Monitoring Core 在 Claude 不在场时的项目级协作规则。用户明确要求：Claude 不在时，主代理完成 Claude 的编排、决策、验收职责；子代理完成编码、文档、QA、诊断设计等执行层任务。

项目级规则无法修改运行时系统提示词，但 `AGENTS.md`、`CLAUDE.md` 和本讨论文档作为仓库协作准则生效。所有代理在 `D:\ai-workbench` 内执行任务时，应以这些文件作为协作、审计和验收依据。

## 核心原则

- 主代理临时代行 Claude 的编排者、决策者、验收者职责。
- 子代理是唯一执行层，负责编码、文档、QA、测试、诊断设计等明确 work unit。
- 主代理不写业务代码是默认硬约束。
- FindX Monitoring Core 默认不使用“主代理单文件小修”例外。
- 子代理必须显式使用 `model: "gpt-5.5"`。
- 子代理 prompt 必须自包含，不依赖隐式上下文。
- 并行子代理必须写集互斥。
- QA FAIL、P0/P1 阻断必须回派修正，不能继续堆功能。
- 主代理不得空等，应在子代理运行期间推进只读审计、验证准备和门禁检查。

## 职责矩阵

| 职责 | 主代理 | 子代理 |
| --- | --- | --- |
| 需求理解 | 负责 | 提供执行视角补充 |
| 任务拆分 | 负责 | 按 work unit 执行 |
| 最终决策 | 负责 | 提供证据和建议 |
| 编码实现 | 默认禁止 | 负责 |
| 文档维护 | 派发和验收 | 负责 |
| QA 测试 | 派发、审查、判定阻断 | 负责执行和报告 |
| 诊断设计 | 派发和验收 | 负责输出方案或配置 |
| 写集互斥 | 负责设计和检查 | 必须遵守 |
| 审计门禁 | 负责 | 提供证据 |
| WSL 验证 | 负责组织和执行 | 可按授权执行 |
| Git 门禁 | 负责 | 不直接越权推进 |
| operations-log 留痕 | 负责或按授权派发 | 可按 work unit 补充 |

## 工作流

1. 主代理读取需求、相关文档和必要代码，确认任务边界、授权路径和风险等级。
2. Claude 不在时，主代理声明接管编排者、决策者、验收者职责，并确认子代理作为唯一执行层。
3. 主代理拆分 work unit，逐项标注角色、允许路径、禁止路径、写集、依赖、验收标准、验证命令、审计门禁和 Git 门禁要求。
4. 主代理派发子代理；所有子代理必须显式设置 `model: "gpt-5.5"`，prompt 必须包含完整上下文和敏感信息占位符规则。
5. 可并行任务必须写集互斥，不得修改同一文件、同一不可分割模块或同一契约面。
6. 子代理执行期间，主代理推进只读审计、风险检查、验证准备或其他不冲突的门禁工作，不空等。
7. 子代理完成后，主代理审查输出、diff、测试证据、风险说明和技术债检查。
8. 执行层缺口、QA FAIL、P0/P1 缺陷、P0/P1 RISK、敏感信息风险必须回派归属子代理修正。
9. 阻断关闭后，主代理同步必要改动到 `/opt/ai-workbench`，按任务类型执行 WSL 验证、浏览器回归、API 验证或文档一致性检查。
10. 主代理完成 Git 门禁：确认 diff 范围、敏感信息、契约变化、数据变化、测试证据、回滚说明和未覆盖项。
11. 主代理输出最终交付摘要，列出修改文件、验证结果、阻断处理、技术债检查和未覆盖风险。

## 禁止事项

- 主代理不得默认直接写业务代码。
- 主代理不得绕过子代理完成编码、文档、QA、诊断设计等执行层任务。
- 主代理不得在 QA FAIL、P0/P1 阻断、敏感信息风险未关闭时继续堆功能。
- 子代理不得越权修改未授权路径。
- 并行子代理不得修改同一文件、同一不可分割模块或同一契约面。
- 不得写入真实密钥、认证票据、Cookie、完整连接串、会话 ID、SSH key。
- 不得修改 `C:\Users\十七\.claude` 下的敏感配置、会话、历史、项目状态文件。
- 不得用文档声明替代真实验证；无法执行验证时必须标记 BLOCKED 或 NOT_RUN。
- 不得把 API_CONTRACT_CHANGE、DATA_CHANGE、权限变化、迁移变化隐含在普通改动中。

## 理论例外

仅在同时满足以下条件时，主代理理论上可直接执行紧急修正：

- 单文件变更；
- ≤10 行核心逻辑；
- 无跨层影响；
- 无 API_CONTRACT_CHANGE；
- 无 DATA_CHANGE；
- 无权限、认证、远程执行、文件上传、WebHook、AI/模型配置风险；
- 能立即完成等价验证；
- 已在 operations-log 留痕，说明未派发原因、风险、验证结果和回滚方式。

FindX Monitoring Core 当前默认不使用该例外。即使满足理论条件，也优先派发给对应子代理完成执行层工作。用户显式授权主代理直接修改文档或协作规则文件时，主代理只能在授权写入范围内执行，不能扩展到业务代码。

## 验证门禁

### 通用门禁

- diff 只包含授权范围内文件。
- 未出现真实 `<TOKEN>` 以外的密钥、Cookie、DSN、SSH key 或会话 ID。
- 所有新增或修改规则使用简体中文 Markdown。
- 子代理输出包含技术债检查：重复逻辑、复杂度、边界、依赖、兼容、测试、回滚和遗留风险。
- 主代理最终输出包含修改文件、摘要、验证结果和未覆盖项。

### 代码变更门禁

- 后端变更：同步到 `/opt/ai-workbench` 后，在 WSL `/opt/ai-workbench/api` 执行 `go build -o api-linux .` 或等价脚本。
- 前端变更：同步到 `/opt/ai-workbench` 后，在 WSL `/opt/ai-workbench/web` 执行 `npm run build` 或等价脚本。
- UI 变更：优先用 Playwright/MCP 做真实浏览器验证，覆盖正常、异常、边界路径。
- API 变更：使用 curl 或测试脚本验证状态码、响应结构、认证、输入校验、错误提示和脱敏。
- QA FAIL、P0/P1 阻断关闭前，不得进入 Git 门禁。

### 文档变更门禁

- 确认文档中的命令、路径、端口、接口清单与代码或既有约定一致。
- 文档-only 变更不跑构建，但需要说明未运行构建的原因。
- 如果文档描述运行规则，必须说明项目级规则无法修改运行时系统提示词。

## 当前 FindX Monitoring Core 执行方式

- 默认由主代理负责 Claude 不在时的编排、决策、验收、审计门禁、WSL 验证和 Git 门禁。
- 默认由子代理负责所有执行层工作：`go-backend`、`vue-frontend`、`qa-tester`、`doc-writer`、`ops-diagnostician`。
- 默认所有可派发子代理使用 `model: "gpt-5.5"`。
- 默认主代理不写业务代码，不使用理论小修例外。
- 默认 QA FAIL、P0/P1 阻断、敏感信息风险、契约或数据变更未验证时，必须回派修正。
- 默认主代理在子代理运行期间不空等，继续推进只读审计、验证准备和门禁检查。

## 主代理评分审计

Claude 缺席时，主代理必须对每个稳定切片给出审计评分。评分不是替代 QA，而是合并与 Git 门禁的输入；QA FAIL、P0/P1 RISK、敏感信息风险和未验证契约变化仍然一票阻断。

| 维度 | 权重 | 审计问题 | 阻断条件 |
| --- | --- | --- | --- |
| 功能正确性 | 30% | 正常、异常、边界、权限、断连路径是否有证据？ | 核心路径失败、状态机错误、误报成功、数据写入不一致。 |
| 代码质量 | 20% | 是否符合 handler/service/store/model/security 或 page/component/api/composable/router 分层？ | God Function、复制粘贴大段逻辑、吞 error、跨层调用。 |
| 安全治理 | 20% | 认证、授权、脱敏、审计、远程执行、AI prompt 是否受控？ | 真实密钥泄露、越权写入、远程执行绕过审批、日志回显敏感信息。 |
| 验证闭环 | 20% | WSL 构建、API/UI 回归、QA 报告和断连验证是否完整？ | 未运行且未标记 BLOCKED/NOT_RUN，或 QA 证据不足却写 PASS。 |
| 文档同步 | 10% | API_CONTRACT_CHANGE、DATA_CHANGE、测试基准、Runbook 是否同步？ | 契约或数据变化未标记，文档与代码入口不一致。 |

评分结论：

- `>= 90` 且无阻断项：可进入 Git 门禁。
- `80-89`：需主代理复核风险并明确接受或回派。
- `< 80`：退回归属子代理修正。
- 存在任一 P0/P1 阻断：无论总分多少均退回。

主代理审计输出必须包含：切片名称、涉及文件、API_CONTRACT_CHANGE、DATA_CHANGE、测试证据、评分表、阻断项、回派对象、回滚方式和未覆盖风险。

## 实时 Git 稳定切片策略

FindX Monitoring Core 采用“稳定切片立即落库”的 Git 策略，避免大批量半成品长期堆在工作区。

稳定切片定义：

- 范围清晰：只覆盖一个可验证功能域，例如 P0-1 target/heartbeat、P0-2 alert rule、P0-3 query gateway、P1 dashboard、P2 agent inspector、P3 remediation precheck。
- 写集清晰：子代理输出不与同批其他子代理修改同一文件、同一不可分割模块或同一契约面。
- 验证清晰：本地验证、WSL 构建或等价验证、API/UI/QA 证据齐全。
- 风险清晰：敏感信息扫描、权限路径、断连路径、回滚方式和未覆盖项已记录。

提交规则：

- 通过本地验证和 QA 的稳定切片立即 commit/push。
- QA FAIL、P0/P1 RISK、敏感信息风险、契约或数据变更未验证时不提交。
- 文档-only 切片可不跑 Go/Vue 构建，但必须说明未运行构建的原因，并完成敏感信息扫描与路径范围检查。
- 每个 commit 信息必须说明阶段、功能域、验证结果和回滚点。
- push 前主代理必须复查 `git diff`、`git status`、敏感信息、授权路径和未跟踪文件，不能把无关改动混入切片。

推荐切片顺序：

1. P0-1 target、datasource 基础语义、agent heartbeat QA 关闭。
2. P0-2 alert rule、current/history event、tryrun、rollback、action QA 关闭。
3. P0-3 datasources、query、query-range、metrics、labels、label-values 查询网关。
4. P1 dashboard/template/evaluator/notification/oncall/silence/subscription/permission/audit。
5. P2 findx-agents：Categraf 插件生态 + Catpaw 授权衍生 inspector。
6. P3 AI 问诊 + 自动修复 precheck/dry-run/approve/execute/verify/rollback/audit。

## 敏感信息占位符

描述敏感配置时只能使用占位符，例如：

- `<API_KEY>`
- `<TOKEN>`
- `<DB_DSN>`
- `<LOGIN_USER>`
- `<SSH_KEY>`
- `<COOKIE>`

不得在 prompt、文档、日志、测试报告或提交信息中写入真实敏感信息。
