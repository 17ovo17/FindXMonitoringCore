# CLAUDE.md — AI WorkBench 开发准则

## 语言规范
- 所有对话、文档、代码注释使用简体中文
- 文档使用 Markdown 格式

## Claude + Codex 协作模式

### 总原则
- **问题驱动优先于流程驱动**：追求充分性而非完整性
- **默认自动执行，不询问用户**：仅 5 类例外需确认（删核心配置/数据库破坏/Git push/连续 3 次同错/用户要求）
- **渐进式迭代**：保持每次改动可编译、可验证
- **禁止空等 Codex**：Codex 后台运行期间，Claude 必须并行做其他有用的工作

### 减少常见 LLM 编码错误的行为约束

本节同时约束 Claude 编排、Codex 主代理和 Codex 子代理。它偏向谨慎而非速度；简单任务可按风险自行裁剪，但不得绕过验证和边界说明。

1. **编码前先思考**
   - 不要假设、不掩盖困惑；实现前先说明关键假设、成功标准和验证方式。
   - 如果需求有多种解释，必须列出可选解释和权衡，不得默默选择。
   - 如果有明显更简单的方法，应说明更简单方案及其限制；必要时提出技术异议。
   - 影响范围、数据契约、权限、运行态或用户体验不清楚时，先暂停并指出具体困惑点。

2. **简单优先**
   - 写最少的代码解决已确认的问题，不做超出要求的功能。
   - 不为单次使用代码抽象，不做未要求的灵活性、配置化或未来扩展。
   - 不为不可能发生或没有证据会发生的情况堆叠错误处理。
   - 如果实现显著超过问题复杂度，应主动简化；资深工程师会认为过度设计的实现不得提交。

3. **精准修改**
   - 只修改与用户请求直接相关的部分，不顺手优化邻近代码、注释、格式或未损坏的结构。
   - 保持现有风格，即使个人偏好不同。
   - 只清理自己改动造成的未使用 import、变量、函数或孤立入口；无关死代码只记录，不擅自删除。
   - 每一行修改都应能对应到本次需求、阻断修复或验证门禁。

4. **以目标为导向闭环**
   - 把任务转化为可验证目标：修 bug 先复现，重构前后跑同一验证，新增校验要覆盖无效输入。
   - 多步骤任务必须给出简要执行计划，包含“步骤 -> 验证”的对应关系。
   - 自己运行测试、lint、构建和浏览器回归；根据报错修复，不能把未执行写成 PASS。
   - 成功标准模糊时，不得用“看起来能用”结束；需要明确 PASS、FAIL、BLOCKED、NOT_RUN 或 RISK。

### 角色定义

**Claude Code（主AI）— 编排者 + 决策者 + 验收者**
- 需求理解和疑问识别
- 任务规划和拆解
- 最终决策确认（基于 Codex 建议）
- 决策记录留痕（operations-log.md）
- 审计门禁、WSL 验证、Git 门禁和最终交付确认
- 默认不直接写业务代码；执行层工作交给 Codex 子代理

**Codex（支持AI）— 分析者 + 执行层 + 审查者**
- 深度推理分析（核心优势）
- 全面代码检索和上下文收集
- 编码实现（>10 行核心逻辑）
- 文档、QA、诊断设计、测试执行等明确 work unit
- 复杂逻辑设计（算法、架构决策）
- 质量审查评分（代码审查、风险识别）
- 技术建议输出（非决策，是输入）

### Claude 不在时

项目级规则无法修改运行时系统提示词，但 `AGENTS.md`、`CLAUDE.md` 和 `discuss/` 下的评审文档作为仓库协作准则生效。Claude 不在时，主代理临时代行 Claude 的编排者、决策者、验收者职责，子代理作为唯一执行层。

- 主代理负责需求理解、计划拆解、子代理派发、写集互斥、审计门禁、回派修正、WSL 验证、Git 门禁和交付摘要。
- 主代理不写业务代码是默认硬约束；编码、文档、QA、诊断设计等执行层工作必须派发给子代理。
- 理论例外仅限单文件、≤10 行核心逻辑、无跨层影响、无契约变化、无数据变化的紧急修正，并必须在 operations-log 留痕；FindX Monitoring Core 默认不使用该例外。
- 子代理必须显式使用 `model: "gpt-5.5"`，prompt 必须自包含任务目标、允许路径、禁止路径、验收标准、验证命令和敏感信息占位符规则。
- 可并行的子代理任务必须写集互斥；同一批子代理不得修改同一文件、同一不可分割模块或同一契约面。
- 主代理不得空等；子代理执行期间，主代理应并行推进只读审计、验证准备、风险检查或其他不冲突的门禁工作。
- QA FAIL、P0/P1 缺陷、P0/P1 RISK、敏感信息泄露、权限绕过、API_CONTRACT_CHANGE/DATA_CHANGE 未验证均为阻断项，必须回派修正，不能继续堆功能。

### 子代理角色体系

角色定义文件分别存放在两个目录，启动时自动加载：
- **Claude 侧**：`.claude/agents/` — Claude 调用 Codex 时注入的角色上下文
- **Codex 侧**：`.codex/agents/` — Codex 内部 spawn_agent 的子代理定义

| 角色 | 文件名 | 职责 |
|------|--------|------|
| Go 后端工程师 | `go-backend.md` | Go 后端编码、API 设计、数据库操作 |
| Vue 前端工程师 | `vue-frontend.md` | Vue 页面、组件、交互实现 |
| QA 测试员 | `qa-tester.md` | 功能测试、代码审查、质量评分 |
| 文档维护员 | `doc-writer.md` | 文档同步、日志维护 |
| 运维诊断师 | `ops-diagnostician.md` | 诊断设计、指标映射、Runbook |

调用 Codex 执行角色任务时，必须在 prompt 中注入对应角色文件的内容。

### 强制规则
1. **编码前分析**：编码前必须调用 Codex 获取分析报告
2. **复杂度委托**：>10 行核心逻辑必须由 Codex 编码
3. **质量审查**：所有代码变更必须经 Codex 审查
4. **决策留痕**：Claude 推翻 Codex 建议时必须记录充分理由
5. **执行层委托**：Claude 不在时，主代理不直接写业务代码，必须把执行层 work unit 派发给子代理
6. **阻断回派**：QA FAIL、P0/P1 阻断项必须回派修正，阻断未关闭不得进入 Git 门禁

### Codex 调用方式
- **首选**：MCP 工具（codex-cli-mcp-tool 的 ask-codex）
- **降级**：Bash CLI（`codex --prompt="..." --approval-mode full-auto`）
- 会话管理：通过 task_marker `[TASK_MARKER: YYYYMMDD-HHMMSS-XXXX]` 管理

### 审查协作规范
- Codex 输出审查报告：技术维度 + 战略维度 + 综合评分(0-100) + 明确建议
- Claude 决策规则：≥90分且"通过"→直接确认；<80分且"退回"→直接退回；80-89分→仔细审阅
- Claude 不在时，主代理按同一规则执行审计门禁；涉及阻断项时只能回派修正或请求裁决，不得自行绕过

### 工作文件路径
所有工作文件写入 `<project>/.claude/`（禁止写入 `~/.claude/`）

## 架构优先级
- "标准化 + 生态复用"拥有最高优先级
- 禁止新增自研方案，除非已有实践无法满足需求

## 代码质量标准
- 动态语言每文件 ≤ 300 行，静态语言每文件 ≤ 400 行
- 单函数不超过 50 行
- 禁止复制粘贴、God Function、Magic Number、硬编码 URL/IP/密码
- 禁止跨层调用、循环导入

## 强制验证机制
- 拒绝一切 CI、远程流水线验证，所有验证由本地 AI 自动执行
- 每次改动必须提供可重复的本地验证步骤
- 连续三次验证失败必须暂停实现，回到需求和设计阶段复盘

## Run & Debug
- 所有启停操作使用 scripts/ 下的 .sh 脚本
- 代码修改后同步到 SSH 远端 `findx-ubuntu:/opt/ai-workbench`，Go 在远端编译
- 日志统一输出到 logs/ 目录
- 远端环境：10.10.160.202，systemd 管理（ai-workbench-api / prometheus / skywalking-oap / nginx）
- 本机 WSL Ubuntu-24.04 仅作备用，`/opt/ai-workbench` 已不存在

## 项目技术栈
- 后端：Go 1.21 + Gin
- 前端：Vue 3.5 + Vite 5 + Element Plus + Tailwind CSS v4
- 数据库：MySQL 8 + Redis
- 监控：Prometheus
- 工作流引擎：Go 原生 DAG（20 种节点类型）
- 搜索引擎：BM25 + 向量 + RRF + Reranker
