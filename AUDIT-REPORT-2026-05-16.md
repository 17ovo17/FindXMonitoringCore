# FindX 平台全面审计报告

> 审计日期：2026-05-16
> 对比源码：Nightingale(n9e) + SkyWalking + SignOz + Categraf + Catpaw
> 对比目标：https://8.lwops.cn (LWOPS 运维智能体)
> 逆向数据：D:\测试\ 目录下的 playwright snapshot + 截图

---

## 一、总体评价

| 维度 | 评分 | 说明 |
|------|------|------|
| 架构骨架 | ★★★★☆ | 路由体系、模块划分、双模存储设计合理 |
| API 覆盖面 | ★★★☆☆ | 路由定义齐全但很多 handler 是空壳 |
| 后端业务深度 | ★★☆☆☆ | 核心引擎（告警评估、通知发送、数据源代理）缺乏真实实现 |
| 前端功能完整度 | ★★★☆☆ | 页面框架和导航齐全，交互深度不足 |
| 前端 UI 层级（抽屉/弹窗） | ★★☆☆☆ | 缺少 2 级/3 级抽屉、全屏编辑器 |
| CMDB 对齐度 | ★★☆☆☆ | 模型/实例框架有，字段和交互远不及 LWOPS |
| Agent 下发能力 | ★☆☆☆☆ | 路由存在但无真实下发引擎 |
| 测试覆盖 | ★★☆☆☆ | 部分单元测试，E2E 极少 |

**一句话：骨架 80%，肌肉 30%，皮肤（交互细节）15%。**

## 六、Agent 远程下发功能设计（待实现）

### 6.1 Categraf 下发（基于 n9e 65 个集成模板）

**流程**：
```
选择集成模板（如 MySQL）
→ 填写连接参数（address/username/password）
→ 渲染 .toml 配置
→ 选择目标主机（从 CMDB）
→ SSH 推送到 /opt/categraf/conf/input.mysql/mysql.toml
→ 执行 systemctl reload categraf
→ 验证 mysql_* 指标到达 Prometheus
```

**n9e 集成模板结构**（每个组件）：
```
integrations/MySQL/
├── collect/mysql/mysql.toml    ← Categraf 采集配置模板
├── dashboards/*.json           ← 预置仪表盘
├── alerts/*.json               ← 预置告警规则
├── metrics/categraf-base.json  ← 指标说明（名称/单位/描述）
├── markdown/README.md          ← 接入文档
└── icon/                       ← 图标
```

**后端需要实现**：
1. 模板渲染引擎（.toml 占位符 → 用户参数）
2. SSH 文件推送（已有基础 remote/exec）
3. 配置版本管理 + 回滚
4. 数据到达验证（查询 Prometheus 确认指标上报）
5. 65 个集成模板数据导入

**前端需要实现**：
1. 集成选择页（65 个组件卡片 + 搜索/分类）
2. 配置表单（根据 .toml 动态生成）
3. 目标主机选择（从 CMDB + 凭证绑定）
4. 下发进度步骤条
5. 下发历史 + 回滚

### 6.2 Catpaw 巡检下发

```
选择巡检插件（CPU/内存/磁盘/Redis/DNS 等 35+）
→ 配置巡检参数
→ SSH 安装 catpaw（如未安装）
→ 推送插件配置到 /opt/catpaw/conf.d/
→ 验证巡检数据上报
```

### 6.3 SkyWalking Agent 下发

```
选择语言（Java/Go/Python/Node.js/.NET）
→ 生成 Agent 配置（collector.backend_service + service_name）
→ SSH 推送 Agent 包
→ 修改应用启动脚本注入 -javaagent
→ 验证 Trace 数据到达
```

## 七、Codex 做得好的地方

1. **架构骨架搭建快** — 路由体系、模块划分、双模存储设计合理
2. **工作流引擎** — 自研 DAG + 35 个内置工作流（源码没有的创新）
3. **知识库 + 搜索引擎** — BM25 + 向量 + RRF 融合（独有能力）
4. **Evidence Chain** — 证据串联（AIOps 创新）
5. **CMDB 导航树** — 完整对齐 LWOPS 菜单结构
6. **API 覆盖面广** — 几乎所有端点都有路由定义

---

## 八、Codex 做得不好的地方

1. **空壳 Handler** — 大量路由只有 CRUD 壳，无真实业务逻辑
2. **前端无抽屉层级** — 所有编辑都是简单 Modal，缺少 2 级/3 级抽屉
3. **CMDB 字段不全** — 没有按资源类型定义专属字段
4. **无 Agent 下发引擎** — 路由存在但无模板渲染和推送逻辑
5. **集成模板数据为空** — 有框架但 65 个组件的实际数据未导入
6. **前端纯 JS 无类型** — 无 TypeScript，无 ESLint，无测试
7. **告警引擎完全缺失** — 最核心的定时评估循环没有实现

## 十、结论

FindX 平台在**广度**上做得不错（路由覆盖 80%，导航树完整对齐 LWOPS），但在**深度**上严重不足：

- 后端最核心的三个引擎（告警评估、通知发送、Agent 下发）都是空壳
- 前端缺少 2 级/3 级抽屉交互体系，所有编辑都是简单弹窗
- CMDB 有框架但字段不全，缺少按资源类型的专属属性和 Agent 状态联动
